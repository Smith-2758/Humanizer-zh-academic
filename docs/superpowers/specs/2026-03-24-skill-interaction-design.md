# Humanizer-zh-academic Skill 交互与规则重构设计

日期：2026-03-24

## 1. 背景

当前仓库本质上是一个基于 `SKILL.md` 的中文学术文本改写 skill。它的核心能力已经比较明确：识别并清理学生学术文本中的 AI 痕迹，将内容改写为更朴实、严谨、自然的学生写作风格。

现阶段的主要问题不在于“规则不够多”，而在于“改写入口不够可控”。skill 目前更像一段静态提示词：用户贴入文本后，模型容易直接开始改写，但无法稳定区分不同写作场景，也无法在输出前确认用户实际需要的写作姿态。

这会导致两个直接问题：

1. 同一段文本在不同场景下应有不同写法，但当前 skill 缺少前置参数采集，输出容易失准。
2. 用户无法直接控制“严肃程度”和“角色定位”，需要通过自然语言临时描述，实际体验不稳定。

本次设计的核心目标，是将该 skill 从“静态改写规则集合”升级为“轻交互式学术改写 skill”。

## 2. 目标与非目标

### 2.1 目标

本次重构需要完成以下目标：

1. 在改写前自动询问并确认 `语气严肃度`。
2. 在改写前自动询问并确认 `角色`。
3. 让 `角色` 成为真正影响写作策略的结构化参数，而不是仅用于装饰的名称。
4. 允许 `自定义角色`，但必须通过受控映射约束用户输入，避免人设自由发散。
5. 当用户已经在原始请求中给出明确要求时，skill 不应重复追问。
6. 保持整体交互尽量短，避免把 skill 变成冗长问卷。

### 2.2 非目标

本轮不处理以下能力：

1. 不新增 `改写强度` 参数。
2. 不引入数据库、语料库或自动评测系统。
3. 不扩展到非学术场景，如营销文案、博客、社交媒体文风。
4. 不做复杂的多轮角色画像访谈，只保留一次受控补问。

## 3. 核心产品思路

skill 运行时应分为两层：

1. 参数采集层
2. 改写执行层

参数采集层负责确认本次改写所需的最小上下文，仅包含：

1. 严肃度
2. 角色

改写执行层负责将以下三部分合成最终输出：

1. 原始文本
2. 严肃度模板
3. 角色模板

这种结构的重点，不是增加功能数量，而是让改写行为“先定姿态，再动笔”。对于学术文本，这一点比继续堆更多反 AI 规则更重要。

## 4. 运行流程设计

### 4.1 默认前提

skill 默认假设：用户在调用时已经提供待处理文本。

因此，运行流程应优先检查文本是否存在，而不是默认先询问角色。只有在用户未提供文本时，才中断并要求其补充文本。

### 4.2 主流程

建议将 skill 的执行流程固定为以下顺序：

1. 检查用户消息中是否包含待处理文本。
2. 若无文本，提示用户补充原文，不进入后续参数采集。
3. 若有文本，检查用户是否已明确给出严肃度。
4. 若未给出严肃度，先询问：`语气严肃度选低、中还是高？`
5. 在严肃度确认后，检查用户是否已明确给出角色。
6. 若未给出角色，再询问：`角色选课程论文作者、毕业设计作者、实验报告作者、答辩陈述稿作者，还是自定义？`
7. 若选择预设角色，则直接进入改写。
8. 若选择自定义角色，则进入一轮受控补问。
9. 参数齐全后，按模板执行改写。
10. 输出参数确认、改写结果和精简修改日志。

### 4.3 关键约束

为保证交互稳定，流程中必须遵守以下规则：

1. 在严肃度和角色未确认完毕前，不得直接开始改写。
2. 若用户已给出某项参数，只能补问缺失项，不得重复询问。
3. 自定义角色只能补问一轮，不允许进入无限追问。
4. 参数采集阶段不得输出半成品改写结果或预览。
5. 提问文案必须尽量短，避免弱化工具感和执行效率。

## 5. 参数识别与优先级

### 5.1 优先级原则

skill 应优先采纳用户在原始请求中明确表达的写作要求，而不是机械触发固定问答。

参数识别优先级如下：

1. 用户显式要求
2. 文本用途中的高置信度隐含信息
3. 交互式补问

### 5.2 示例

例如：

- 用户说“按毕业设计口吻改，正式一点”，则可直接识别为：
  - 角色：毕业设计作者
  - 严肃度：高
- 用户说“改成答辩稿，别太口语”，则可优先识别为：
  - 角色：答辩陈述稿作者
  - 严肃度：中或高，默认高
- 用户只贴原文，没有任何附加说明，则应按流程补问。

这里的目标不是覆盖所有自然语言表达，而是处理高频、明显、低歧义的要求，减少不必要追问。

## 6. 严肃度设计

本次仅保留三档：

1. 低
2. 中
3. 高

### 6.1 低

定位：自然、朴实、保留适度学生写作痕迹。

表现要求：

1. 句子可以稍微自然，不必过度压缩。
2. 允许少量生活化但不过分口语的连接表达。
3. 保留一定个人书写痕迹，但不能松散、轻浮。
4. 去 AI 的重点在“去卖弄”和“去机械感”，而不是极端正式化。

适用场景：

1. 普通课程作业
2. 平时小论文
3. 一般性的阶段总结

### 6.2 中

定位：克制、清楚、稳妥。

表现要求：

1. 压缩主观色彩和松散表达。
2. 句式更整齐，但仍保留自然节奏。
3. 术语保留更稳，避免过度口语或过度板正。
4. 适合作为默认正式学术表达层级。

适用场景：

1. 课程论文
2. 大多数实验报告
3. 一般性正式提交文本

### 6.3 高

定位：正式、收束、凝练。

表现要求：

1. 严控修饰词与情绪化表达。
2. 减少主观判断型语气。
3. 更强调客观陈述、结构稳定和结论克制。
4. 保留必要术语，但不允许花哨卖弄。

适用场景：

1. 毕业设计正文
2. 正式提交给导师或评阅方的文本
3. 摘要、总结、结论等高正式度内容

### 6.4 严肃度影响维度

严肃度必须影响实际写法，而不仅是一句附加说明。至少应映射到以下维度：

1. 句长波动
2. 第一人称使用比例
3. 主观评价语气强弱
4. 修饰词保留程度
5. 术语保留与降级尺度
6. 结论段收束力度

## 7. 角色模板设计

本轮采用四个预设角色。

### 7.1 课程论文作者

定位：偏分析与论述的学生学术写作。

写法要求：

1. 强调论述清楚、结构完整。
2. 不应过度工程化。
3. 不应写成口头答辩风格。
4. 适合课程论文、综述、小型专题分析。

### 7.2 毕业设计作者

定位：偏正式提交的项目型、实现型写作。

写法要求：

1. 强调方案、实现、验证、问题与改进。
2. 更重视“做了什么、为什么这样做、结果如何”。
3. 允许保留必要技术细节。
4. 结论部分应更加克制、完整。

### 7.3 实验报告作者

定位：偏客观记录与分析的实验型文本。

写法要求：

1. 强调实验目的、步骤、结果、误差、分析。
2. 不追求宏大叙事。
3. 重点删除空泛总结和伪深刻表述。
4. 结果陈述应尽量具体。

### 7.4 答辩陈述稿作者

定位：正式但可讲述的口头表达文本。

写法要求：

1. 强调逻辑分段清楚。
2. 句子要顺口，便于口头呈现。
3. 信息密度不能过高，避免一口气塞满。
4. 可使用适度口头组织句，但不能模板化、舞台化。

### 7.5 角色影响维度

角色模板至少应控制以下写作特征：

1. 叙述重心
2. 段落组织方式
3. 允许的专业度
4. 结尾方式
5. 更偏书面阅读还是口头表达

## 8. 自定义角色的受控映射设计

### 8.1 设计原则

自定义角色不能直接开放成“任意人设描述”，否则用户可能给出模糊、冲突或过度戏剧化的要求，导致输出失控。

因此，自定义角色应被定义为“受控画像输入”，其目标是把用户需求重新映射回 skill 可处理的学术写作画像。

### 8.2 补问方式

当用户选择 `自定义` 后，skill 只进行一次补问，固定采集以下三项信息：

1. 学科背景
2. 求学阶段或经验
3. 文本用途

建议提问文案保持极短，例如：

`请补充这三个信息：学科背景、求学阶段或经验、文本用途。`

### 8.3 三项信息的作用

`学科背景`

作用：

1. 决定术语保留尺度。
2. 决定表达是否偏工程、偏分析、偏论述。

`求学阶段或经验`

作用：

1. 决定正式度和成熟度。
2. 避免把本科课程作业写成研究生论文语气，或反过来过于稚嫩。

`文本用途`

作用：

1. 决定段落组织方式。
2. 决定是否偏阅读文本还是讲述文本。

### 8.4 内部映射要求

收到这三项信息后，skill 不应原样照搬用户描述，而应在内部将其映射回一个受控写作画像：

1. 基于学科背景调整术语密度。
2. 基于求学阶段调整正式程度和表达老练度。
3. 基于文本用途选择最接近的角色写法。

换句话说，自定义角色的“开放性”只存在于输入层，输出层仍然必须受规则控制。

## 9. 改写执行规则

参数采集完成后，改写应遵循以下优先级：

1. 原有通用去 AI 规则
2. 严肃度模板规则
3. 角色模板规则
4. 用户在请求中的特殊要求

### 9.1 保留的现有核心能力

本轮不推翻当前 skill 的核心定位，仍保留以下主规则：

1. 去学术卖弄
2. 拆机械句法
3. 降低空洞宏大叙事
4. 限制 AI 高频词
5. 清洗破折号、粗体和内联标题等高危排版痕迹

### 9.2 新增执行原则

在原有规则基础上新增以下执行原则：

1. 先按角色确定写法重心，再按严肃度调整正式程度。
2. 若角色与严肃度冲突，以角色场景优先、严肃度微调为辅。
3. 若用户明确要求与默认模板冲突，以用户要求优先，但不得偏离学术写作定位。

## 10. 输出格式设计

默认输出建议固定为三部分：

1. 参数确认
2. 重写后的文本
3. 精简修改日志

### 10.1 参数确认

内容应尽量短，例如：

- 严肃度：高
- 角色：毕业设计作者

参数确认的目的，是让用户明确知道本次输出基于什么姿态生成。

### 10.2 重写后的文本

这是主要输出，必须完成以下目标：

1. 更像真实学生写作
2. 降低机械感和卖弄感
3. 符合已确认的严肃度与角色

### 10.3 精简修改日志

修改日志仅保留高价值说明，建议包括：

1. 删除或压缩了哪些套话、卖弄表达
2. 调整了哪些句法节奏问题
3. 根据角色做了哪些语气和结构适配

## 11. 边界与异常处理

### 11.1 应避免的行为

skill 不应出现以下行为：

1. 用户未给齐参数时直接改写。
2. 用户已经说明参数时重复追问。
3. 自定义角色阶段反复追加问题。
4. 在学术文本场景中输出过度营销化或强行口语化结果。
5. 将角色概念演变为无边界人物设定。

### 11.2 异常处理

建议处理方式如下：

1. 若用户未提供原文，则提示其先补充文本。
2. 若用户只提供了部分参数，则只补问缺失项。
3. 若用户的角色描述和文本用途明显冲突，则以文本用途优先解释。
4. 若用户要求明显偏离 skill 定位，应将输出拉回学术文本范围内。

## 12. 对 `SKILL.md` 的重构建议

建议将后续 `SKILL.md` 重构为以下结构：

1. Skill 定位与总目标
2. 运行前置规则
3. 参数采集流程
4. 严肃度映射表
5. 角色模板表
6. 自定义角色受控采集规则
7. 通用去 AI 改写规则
8. 输出格式
9. 边界与禁止行为

这样做的优势是：

1. 让模型先知道什么时候该问。
2. 再知道问完以后如何根据参数改写。
3. 将原有大量规则从“平铺罗列”转为“按执行顺序组织”。

## 13. 验收标准

本次重构完成后，应至少满足以下标准：

1. 用户只贴原文时，skill 会先采集参数，而不是直接改写。
2. 用户已明确说明参数时，skill 不会重复提问。
3. 同一段文本在三档严肃度下会出现明显但不过度的写法差异。
4. 同一段文本在四种角色下会体现不同写作重心。
5. 自定义角色不会演变为无法控制的人设扩写。
6. 默认输出简洁、可读，不显得像评测系统。

## 14. 后续实施建议

建议后续实施分两步进行：

1. 先按本设计重写 `SKILL.md` 的整体结构和交互规则。
2. 再用少量真实样本文本回测不同角色与严肃度是否稳定生效。

本轮不要求开发独立工具链，只要保证 skill 本体的交互逻辑和规则映射稳定即可。
